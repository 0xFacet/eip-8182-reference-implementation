# EIP-8182 Reference Implementation (Groth16/BN254 + split-proof)

Reference implementation of [EIP-8182 — Native Shielded Pool](https://eips.ethereum.org/EIPS/eip-8182), tracking the post-pivot Groth16/BN254 + split-proof variant of the spec. Includes circuits, system contract, on-chain verifier wrappers, helper library, and the asset-bundle generator the EIP links to.

This is a **reference implementation**: not audited, not production-ready, and not optimized. Its purpose is to make the protocol concretely buildable, to keep the EIP's normative artifacts (verification key, precompile vectors, genesis state dump) in sync with the spec, and to give wallet/protocol developers a complete starting point.

## Asset bundle (what the EIP links to)

The spec at `EIPS/eip-8182.md` references these asset files under `assets/eip-8182/`. Three groups:

| File | What it pins | Generated by |
|---|---|---|
| `pool_vk.bin` (1856 B) | Canonical Groth16 verification key (Section 5.5; α∥β∥γ∥δ∥IC[0..21] in EIP-196/197 byte layout) | `scripts/assets/vk_to_bin.js` |
| `pool_vk.sha256` | SHA-256 of the above; clients verify the embedded VK matches at install time | `scripts/assets/refresh.sh` |
| `pool_precompile_happy_path.json` | Valid (proof, 21 publicInputs) — precompile MUST return `uint256(1)` | `scripts/assets/gen_precompile_vectors.js` |
| `pool_precompile_invalid_proof.json` | Bit-flipped C.x — precompile MUST return `uint256(0)` and MUST NOT revert | same |
| `pool_precompile_noncanonical_field.json` | publicInputs[0] = p — precompile MUST reject (Section 3.5) | same |
| `shielded-pool-state.json` | Genesis state dump installed at `SHIELDED_POOL_ADDRESS` at activation fork (Section 5.1) | `contracts/script/InstallSystemContracts.s.sol` |
| `poseidon2_bn254_t4_rf8_rp56.json` | Round constants + internal diagonal | hand-curated, never regenerated |
| `poseidon2_vectors.json` | Cross-impl test vectors for the sponge | hand-curated |
| `delivery_scheme1_vectors.json` | Companion-standard delivery scheme 1 (ML-KEM-768 / FIPS 203) test vectors | hand-curated |

Refresh everything in one command:

```bash
EIP_ASSETS_DIR=/path/to/EIPs/assets/eip-8182 ./scripts/assets/refresh.sh
```

Without `EIP_ASSETS_DIR` the bundle stays in `assets/eip-8182/` for local inspection.

## Build prerequisites

- Node 20+
- Foundry (`forge`, `anvil`)
- The vendored `vendor/circom` works on macOS arm64; on other targets, install circom 2.2.3 and point `CIRCOM` at it
- `jq` (used by the install script and the asset refresh)
- ~700 MB of disk + ~5 GB of RAM for the snarkjs trusted setup

```bash
npm install
./scripts/circuit/build_pool.sh   # one-time: compile + dev-MPC trusted setup
forge build
```

The included `build/pool/pool_final.zkey` is a single-contributor dev key. Production deployments MUST replace it with the output of a real multi-party ceremony (see "Trusted setup" below).

## End-to-end demo

```bash
node scripts/integration/build_session.js   # writes build/integration/session.json
forge test                                  # 10/10 should pass
```

`Integration.t.sol` exercises:

- Pool precompile (etched at `0x...0030`) accepts a real proof, rejects a tampered one, rejects a non-canonical public input.
- `DemoAuthVerifier.verifyAuth` accepts a real proof, rejects wrong public inputs.
- Full `transact` flow against mock verifiers — confirms tree updates, root histories, nullifier marking, intent-replay-id consumption, and event emission.
- `registerUser`, `registerAuthPolicy`, `deregisterAuthPolicy` (with cross-user revocation rejected), and `deposit` advance their respective trees.

### Pool proving benchmark

This is a pool-proof-only benchmark. It does not include auth proving, wallet note scanning, note encryption/decryption, RPC, transaction submission, or block inclusion.

The current pool circuit has 223,043 constraints, 21 public inputs, and 808 private inputs. The benchmark witness is the worst-case private-transfer shape for this circuit: two real input notes, three real output notes, private fee output active, and all three output slots locked.

The `build_session.js` timing uses `snarkjs.groth16.prove` under Node:

## What's normative vs reference

**Normative** (defined by the EIP, copied into `assets/eip-8182/` and referenced from the spec):
- `pool.circom` and the four common modules (`poseidon2_perm`, `poseidon2_sponge`, `merkle`, `components`, `bits`, `domain_tags`). The constraint relation MUST match the spec's Section 9 description.
- `pool_vk.bin` and its sha256 — pinned by `EIPS/eip-8182.md` Section 5.5.
- The shielded-pool genesis state dump.

**Non-normative** (the EIP only requires that *some* contract implementing the interface exists at `SHIELDED_POOL_ADDRESS`, not this specific Solidity):
- `ShieldedPool.sol` itself — any equivalent contract works.
- `MockPoolPrecompile.sol` — only used in tests; the real precompile is implemented in client EVM code at fork time.
- `auth-demo/auth_demo.circom` and `DemoAuthVerifier.sol` — Section 14's example, intentionally minimalist (2,112 R1CS constraints, ~0.1s prove). It exercises the split-proof contract integration end-to-end; **it does NOT validate prover feasibility for a real auth circuit**. A production auth method would be ECDSA secp256k1 over an EIP-712 typed-data hash via 0xPARC's `circom-ecdsa` (~1.5M constraints) plus an in-circuit keccak (~150K). Companion ERCs deploy these permissionlessly per Section 12.

## Trusted setup

The included `pool_final.zkey` was produced by `snarkjs groth16 setup` followed by a single contributor (`scripts/circuit/build_pool.sh`). For a real launch:

1. Use a Powers-of-Tau ceremony output of size ≥ 2^19 (224K constraints requires `2^N >= 2*223513`, fits in 2^19).
2. Coordinate a multi-party Groth16 phase-2 ceremony with N independent contributors (50–100 is typical for shielded-pool deployments). Any one honest contributor suffices for soundness.
3. Compute the SHA-256 of the resulting `pool_vk.bin` and pin it in the spec's `pool_vk.sha256` asset.
4. Hard-fork-pin the precompile's embedded VK to that digest.
